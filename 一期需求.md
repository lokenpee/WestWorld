# 需求3：世界书人物随着章节进行不断增加内容，容易爆token
更好的做法（既增量更新又不爆 token）

基于现有架构（LLM 出 patch、本地合并）最稳的方向是：

1. **默认用"并行/独立 + 增量合并"**
   - 让 LLM 只看"章节文本 + 上一章摘要"
   - 输出本章 patch
   - 世界书一致性主要靠本地 merge，而不是把全世界书塞给模型

2. **对"内容重复追加"的控制**
   - 当前增量合并只用"前50字包含判断"
   - LLM 只要改写一点就会继续追加
   - 应升级为"更强的去重/相似度判断/按字段结构化更新"
   - 而不是纯文本 append

# 需求4 api识别人物加入世界书 似乎是每个人物都会识别，并不会判断他是主要任务还是次要人物，还是NPC
你的观察完全正确：目前世界书生成时，AI 只要识别到“人物”就会全部加入世界书，并没有区分主角、配角、NPC。根本原因是——

Prompt 没有要求分类：在系统 prompt 和输出模板里，只要求“识别所有人物”，没有让 AI 按“主/配/NPC”分级。
合并逻辑不做筛选：本地合并时只按条目名合并，不会根据“重要性”做过滤或分组。

解决思路
1. Prompt 级别改进
在系统 prompt 里明确要求：“请将人物分为主角、配角、NPC，并在条目中用 角色类型 字段标注（如：主角/配角/NPC）。”
输出模板举例：
2. 本地筛选/分组
合并后在 UI 或导出时，允许用户按“角色类型”筛选/分组/隐藏 NPC。
可以在世界书面板加个筛选器，只显示主角/配角。
3. 后处理/批量标注
如果历史数据没分好，可以加个“批量标注”工具，让用户手动或用规则（如出现频率）批量标记主配NPC。
4. 更智能的自动判别（进阶）
统计每个人物在小说中出现的频率、章节分布，辅助自动标注主配NPC（可作为后续版本优化）。
