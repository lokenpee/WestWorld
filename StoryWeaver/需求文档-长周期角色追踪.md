# StoryWeaver 需求文档：长周期角色追踪与演变聚合

## 文档信息
- **需求编号**: SW-REQ-001
- **需求名称**: 长周期角色追踪与演变聚合
- **优先级**: P1（高优先级）
- **状态**: 待实现
- **创建日期**: 2026-04-06
- **相关模块**: 世界书合并、条目管理、AI处理

---

## 1. 问题背景

### 1.1 当前痛点
在处理长篇小说（特别是100章以上的作品）时，同一角色/地点/物品会在多个章节反复出现，且信息会随剧情发展不断更新。当前插件采用**简单追加**策略，导致以下问题：

```
【示例】角色"张三"在1000章小说中的处理方式：

第1章："张三，男，剑客，性格豪爽"
第2章："张三，男，剑客，性格豪爽 --- 张三学会了御剑术"
第50章："张三... --- 张三结婚了"
第500章："张三... --- 张三成为了剑圣"
第1000章：内容已累积超过5000字，早期信息被淹没
```

### 1.2 具体影响

| 问题 | 严重程度 | 影响描述 |
|------|---------|---------|
| **Token超限** | 🔴 严重 | 长条目超出API输入限制，导致处理失败 |
| **信息稀释** | 🟠 高 | 关键设定被淹没在大量重复描述中 |
| **上下文断裂** | 🟠 高 | AI无法有效追踪角色跨章节的演变轨迹 |
| **世界书臃肿** | 🟡 中 | 导出后的世界书难以阅读和使用 |
| **酒馆性能下降** | 🟡 中 | 超长条目增加SillyTavern的Token计算负担 |

---

## 2. 需求目标

### 2.1 核心目标
实现**智能化、结构化的条目演变追踪**，让角色/地点/物品的发展轨迹清晰可见。

### 2.2 具体目标

1. **自动摘要**：AI自动总结角色的阶段性变化，而非简单追加
2. **时间线追踪**：记录条目在不同章节的关键变化节点
3. **智能去重**：识别并去除重复描述，保留关键信息
4. **演变可视化**：用户可以查看角色的完整发展轨迹

---

## 3. 用户场景

### 场景1：长篇玄幻/修真小说
**用户**: 导入一部3000章的修真小说
**痛点**: 
- 主角从练气期到渡劫期，经历数十次境界突破
- 当前插件累积了主角的全部信息，条目超过1万字
- Token超限，无法继续处理后续章节

**期望**:
```json
{
  "主角": {
    "基础信息": "张三，男，修真者",
    "当前境界": "渡劫期",
    "演变时间线": [
      {"章节": "第1章", "事件": "练气期入门", "描述": "..."},
      {"章节": "第100章", "事件": "突破筑基", "描述": "..."},
      {"章节": "第1000章", "事件": "金丹大成", "描述": "..."}
    ],
    "当前状态摘要": "张三经过3000章修炼，已达渡劫期巅峰，准备飞升..."
  }
}
```

### 场景2：角色关系追踪
**用户**: 追踪反派角色的黑化历程
**痛点**:
- 反派在第1章是正面角色，第500章黑化，第1000章洗白
- 当前世界书只有累积描述，无法看出性格变化轨迹
- 酒馆AI使用时无法准确把握角色当前立场

**期望**:
- 能清晰看到角色立场变化的时间线
- AI能根据当前剧情阶段输出对应的角色状态

### 场景3：地点变迁追踪
**用户**: 追踪一个地点的发展
**痛点**:
- "青云宗"从第1章的小门派发展到第1000章的顶级宗门
- 当前世界书只有一堆零散描述，没有发展脉络

## 4. 功能需求

### 4.1 需求1：智能条目摘要（AI-powered Summarization）

#### 描述
当条目内容超过阈值（如2000字）时，自动触发AI摘要，将累积内容压缩为结构化摘要。

#### 输入示例
```
【原始内容】
张三，男，剑客，性格豪爽
---
张三在第2章学会了御剑术，师父是李明
---
张三在第50章结婚，妻子是王小红
---
张三在第100章击败了魔王，拯救了世界
---
张三在第500章成为剑圣，收徒100人
...(累计5000字)
```

#### 输出示例
```
【AI摘要后】
【基本信息】张三，男，剑客，性格豪爽
【成长历程】从村庄少年成长为剑圣
【重要关系】师父：李明；妻子：王小红；弟子：100人
【关键成就】击败魔王（第100章）、成为剑圣（第500章）
【当前状态】剑圣境界，隐居中

【时间线】
第1章: 出场，村庄少年
第2章: 拜入师门，学习御剑术
第50章: 与王小红成婚
第100章: 击败魔王，拯救世界
第500章: 突破成为剑圣
```

#### 验收标准
- [ ] 可配置自动摘要触发阈值（字数）
- [ ] 摘要后Token数减少50%以上
- [ ] 保留所有关键信息和章节标记
- [ ] 支持手动触发摘要

### 4.2 需求2：条目演变追踪（Entry Evolution Tracking）

#### 描述
为每个条目建立演变历史，记录每次更新的章节、时间、变更内容。

#### 数据结构
```typescript
interface EntryEvolution {
  entryName: string;
  category: string;
  createdAt: {
    chapterIndex: number;
    timestamp: number;
  };
  evolutionChain: EvolutionNode[];
}

interface EvolutionNode {
  chapterIndex: number;      // 第X章
  timestamp: number;         // 处理时间
  changeType: 'add' | 'update' | 'milestone';  // 变更类型
  summary: string;           // AI生成的变更摘要
  previousState: string;     // 变更前状态（摘要）
  newState: string;          // 变更后状态（摘要）
  fullContent: string;       // 完整内容（存储在DB）
}
```

#### 功能点
1. **自动记录**：每次合并条目时自动记录变更节点
2. **里程碑标记**：AI识别重大变化（如境界突破、死亡、黑化）并标记
3. **时间线查看**：用户可以查看条目的完整演变时间线
4. **版本回退**：可以回退到任意历史版本

### 4.3 需求3：智能变更检测（Intelligent Change Detection）

#### 描述
当前代码只用简单字符串匹配判断是否重复。需要AI智能判断新内容是"补充"还是"更新"。

#### 示例
```
第1章输出："张三，男，25岁，剑客"
第50章输出："张三，男，26岁，剑客，已婚"

当前处理结果：
"张三，男，25岁，剑客 --- 张三，男，26岁，剑客，已婚"
（重复且矛盾：到底是25岁还是26岁？）

期望处理结果：
"张三，男，26岁，剑客，已婚"
（智能合并：更新年龄，补充婚姻状态）
```

#### 实现方案
```javascript
// 新增：AI智能合并模式
async function intelligentMerge(existingEntry, newEntry, chapterIndex) {
  const prompt = `
请合并以下两个时间点的角色信息：

【第${existingEntry.chapterIndex}章信息】
${existingEntry.content}

【第${chapterIndex}章新信息】
${newEntry.content}

要求：
1. 如果信息冲突（如年龄变化），以新信息为准
2. 如果信息互补，合并到一处
3. 保留时间线标记（如"第X章时..."）
4. 去除重复描述

输出格式：
{
  "mergedContent": "合并后的内容",
  "changes": [
    {"field": "年龄", "old": "25", "new": "26"},
    {"field": "婚姻", "old": "未婚", "new": "已婚"}
  ]
}
`;
  const response = await callAPI(prompt);
  return parseMergeResult(response);
}
```

### 4.4 需求4：分卷智能合并（Smart Volume Merging）

#### 问题
当前分卷模式下，同名条目会被简单加上`_卷X`后缀：
```
李尘家别墅
李尘家别墅_卷2
李尘家别墅_卷3
```

#### 改进方案
分卷时自动摘要前置卷的内容，而不是简单保留全部：

```javascript
function createNewVolume() {
  // 当前实现：直接保存完整世界书
  AppState.worldbook.volumes.push({
    volumeIndex: currentIndex,
    worldbook: JSON.parse(JSON.stringify(AppState.worldbook.generated))
  });
  
  // 改进方案：对长条目进行摘要后再保存
  const summarizedWorldbook = summarizeLongEntries(
    AppState.worldbook.generated,
    maxLength: 1000  // 每个条目最多保留1000字
  );
  
  AppState.worldbook.volumes.push({
    volumeIndex: currentIndex,
    worldbook: summarizedWorldbook,
    fullWorldbook: JSON.parse(JSON.stringify(AppState.worldbook.generated))  // 完整版存储
  });
}
```

## 5. 技术方案

### 5.1 架构调整

```
当前架构：
ProcessingService → mergeWorldbookDataIncremental → 直接追加

新架构：
ProcessingService → EntryEvolutionTracker → IntelligentMerger → Worldbook
                          ↓
                   EvolutionHistoryDB (IndexedDB)
```

### 5.2 新增模块

| 模块 | 职责 | 文件 |
|------|------|------|
| EntryEvolutionTracker | 追踪条目演变历史 | services/entryEvolutionService.js |
| IntelligentMerger | AI智能合并条目 | services/intelligentMergeService.js |
| EntrySummarizer | 条目摘要生成 | services/summarizerService.js |
| EvolutionUI | 演变时间线展示 | ui/evolutionView.js |

### 5.3 数据存储

```javascript
// IndexedDB 新增表
{

    { name: 'entryName', keyPath: 'entryName' },
    { name: 'category', keyPath: 'category' },
    { name: 'chapterIndex', keyPath: 'chapterIndex' }
  ]
}
```

## 6. 配置项

### 6.1 新增设置

```javascript
// core/constants.js
const defaultSettings = {
  // ... 现有配置
  
  // 演变追踪配置
  evolutionTracking: {
    enabled: true,              // 是否启用演变追踪
    autoSummarizeThreshold: 2000,  // 自动摘要阈值（字）
    milestoneDetection: true,   // 是否检测里程碑事件
    maxEvolutionNodes: 100,     // 最大演变节点数
    storageMode: 'full' | 'summary' | 'hybrid'  // 存储模式
  },
  
  // 智能合并配置
  intelligentMerge: {
    enabled: true,              // 是否启用智能合并
    conflictResolution: 'newer' | 'ai' | 'manual',  // 冲突解决策略
    preserveTimeline: true      // 是否保留时间线标记
  }
};
```

### 6.2 存储模式说明

| 模式 | 描述 | 适用场景 |
|------|------|---------|
| `full` | 保留完整演变历史 | 短篇/中篇小说 |
| `summary` | 只保留摘要 | 超长篇（>1000章） |
| `hybrid` | 近期保留完整，早期只保留摘要 | 长篇（300-1000章） |
